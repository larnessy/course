        API Security Top 10

Преподаватель отметил в чате, что прям все составляющие не требуются: «В основном
про инъекции и аутентификацию и авторизацию», — так что проведу проверку и обеспечу ключевые пункты.

    A01: Broken Access Control
- Контроль доступа обеспечен
- Небезопасные ссылки на чувствительные данные отсутствуют
    A02: Cryptographic Failures
- Хеширование паролей с использованием соли
- Передаваемые данные содержат только необходимую информацию
- TLS сертификат сказали не реализовывать
    A03: Injection
- Использованы параметризированные запросы и ORM последней версии
- POST запросы закрыты для пользователей с ролью USER
    A04: Insecure design
- Восстановление пароля попросту невозможно
- resilience4j в какой-то степени ограничивает использование API
- Проведены интеграционные тесты
    A05: Security Misconfiguration
- Сообщения об ошибках «стандартизированы» (stack traces возвращаться не будут)
- У приложения более не порт по умолчанию (прочие порты не закрыты за ненадобностью)
- Административные панели отсутствуют
- Usernames and passwords по умолчанию нет
   A06: Vulnerable/Outdated Components
- В настоящий момент системы и библиотеки обновлены
- Об имеющихся в проекте зависимостях, содержащих уязвимости, мне неизвестно
    A07: Identification/Authentication Failures
- Двухфакторную аутентификацию сказали не делать
- JWT не нужен
- Пара ограничений на пароли в учебных целях установлены
- Восстановление не доступно
- Rate limiting в некотором виде присутствует
    A08: Software and Data Integrity Failures
- Нет автообновлений стороннего ПО
- Уязвимость десериализации, на сколько я могу судить, отсутствует
    A09: Insufficient Logging & Monitoring
- Логи и метрики сказали не делать
    A10: Server-Side Request Forgery (SSRF)
- Клиенту не возвращаются грязные данные
- В whitelist нет необходимости
